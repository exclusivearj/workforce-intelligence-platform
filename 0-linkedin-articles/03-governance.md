Data governance documentation that lives in a wiki gets stale. Governance that lives in code stays accurate because it has to — if it drifts from reality, the pipeline breaks.

This module implements governance as code: a YAML classification config drives Python codegen that produces Postgres DDL, masking views, and audit configuration. One edit, one command, everything in sync.

---

**The classification config**

Every column in every table in the platform gets a sensitivity level:

```yaml
tables:
  analytics.dim_employees:
    columns:
      employee_id:        { sensitivity: public,       pii: false }
      department:         { sensitivity: internal,     pii: false }
      full_name:          { sensitivity: confidential, pii: true,  mask_with: "'***'" }
      email:              { sensitivity: confidential, pii: true,  mask_with: "'***@***.com'" }
      salary:             { sensitivity: restricted,   pii: true,  mask_with: "NULL" }
      performance_rating: { sensitivity: restricted,   pii: true,  mask_with: "NULL" }
```

Four levels:

| Level | Meaning | LLM context | Masked in views |
|---|---|---|---|
| `public` | No restrictions | Yes | No |
| `internal` | Authenticated employees only | Yes | No |
| `confidential` | Named roles only | No | Yes (with mask expression) |
| `restricted` | HR Partner + Legal only, audited | No | Excluded entirely |

The config is Pydantic-validated on load. An invalid sensitivity level, a missing `mask_with` expression on a confidential PII column, or a table not in the schema — all raise errors before any DDL is generated.

**Code-generated DDL**

The codegen takes the config and produces three artifacts:

**1. Grant statements**

```sql
-- Generated by ddl_generator.py at 2024-01-15T09:00:00 from config v1.0
GRANT SELECT ON analytics.dim_employees TO analyst_reader;
REVOKE SELECT ON analytics.dim_employees FROM analyst_reader;
GRANT SELECT ON analytics.v_employees_safe TO analyst_reader;  -- masked view instead
```

Restricted columns are revoked on the base table and re-granted via the masked view only. The analyst_reader role never has direct access to salary or performance_rating — not via a query, not via `SELECT *`.

**2. Masking views**

```sql
CREATE OR REPLACE VIEW analytics.v_employees_safe AS
SELECT
    employee_id,          -- public: as-is
    department,           -- internal: as-is
    '***' AS full_name,   -- confidential: masked
    '***@***.com' AS email, -- confidential: masked
    MD5(manager_id::text) AS manager_id_hashed, -- confidential: hashed
    -- salary: EXCLUDED (restricted)
    -- performance_rating: EXCLUDED (restricted)
    level,
    hire_date,
    is_active
FROM analytics.dim_employees;
GRANT SELECT ON analytics.v_employees_safe TO analyst_reader;
```

**3. The workflow for adding a new sensitive column**

1. Add the column to `data_classification.yml` with sensitivity + mask_with
2. `make generate-ddl` — regenerates the SQL scripts in `generated/`
3. Review the diff (the generated SQL is committed, so `git diff` shows exactly what changed)
4. `make apply-ddl` — applies to Postgres
5. `make test` — asserts the column is handled correctly in views and grants

One YAML edit. No hunting through SQL files. No risk of forgetting to update a view.

**The audit layer**

True column-level SELECT auditing in Postgres requires either the `pg_audit` extension or application-layer enforcement. Standard Postgres doesn't support column-level SELECT triggers.

I chose the honest approach: a weekly Airflow DAG that scans `pg_stat_statements` for queries containing restricted column names, checks whether the executing role is in the allowed list, and flags violations.

This is not foolproof — it's text matching against query strings, not cryptographic proof of access. The README documents this limitation explicitly. In a production environment, you'd add `pg_audit` at the Postgres level. But for a portfolio project, the honest documentation of a trade-off is more valuable than a fake solution that papers over the limitation.

**The data access matrix**

The `policies/data_access_matrix.md` is written by hand, not generated. It encodes policy decisions — who should have access to what, and why — that require human judgment and sign-off from Legal and HR.

A generated access matrix signals that no human reviewed the policy. A hand-written, version-controlled matrix with a named owner and a last-updated date is what a compliance reviewer actually wants to see.

```markdown
## Column access by sensitivity

| Sensitivity | analyst_reader | dbt_transformer | hr_partner_role | legal_role |
|---|---|---|---|---|
| public | ✓ | ✓ | ✓ | ✓ |
| internal | ✓ | ✓ | ✓ | ✓ |
| confidential | masked view | ✓ | ✓ | ✓ |
| restricted | excluded | excluded | ✓ (audited) | ✓ (audited) |
```

**The weekly governance DAG**

```
scan_audit_log
      │
      ▼
flag_unexpected_access   ←── any analyst_reader touching restricted columns?
      │
      ▼
send_weekly_report       ←── Slack/email to data platform team
```

The report is a Markdown summary: total queries scanned, findings by severity, top offending roles. It runs Sunday at 3am, before the Monday work week.

**What this module demonstrates**

Most data governance implementations I've seen are either a Confluence page nobody updates or a one-time SQL script someone ran two years ago. Neither gives you confidence in what the system is actually doing.

Code-generated governance, version-controlled config, and automated audit scanning give you three things: auditability (what the policy is, as code), enforceability (DDL that actually applies the policy), and observability (a DAG that checks whether it's being followed).

That combination is what "data governance requirements for employee or sensitive data" means in a production system.
